Chronos-2 · Forecasting Architecture

Amazon's 120 M-parameter encoder-only foundation model for univariate, multivariate, and covariate-informed probabilistic forecasting.

Chronos-2 takes grouped time-series — targets, past covariates, and known future covariates — robustly scales them, slices them into 16-step patches, and passes them through 12 encoder blocks that alternate time self-attention along the temporal axis and group self-attention across related series. A direct multi-step quantile head emits 21-quantile forecasts up to 1024 steps ahead in a single forward pass — no autoregressive token sampling.

02 · Architecture

End-to-end inference pipeline

Raw pandas-style tables → grouped, scaled, patched tensors → 12 encoder blocks → 21-quantile direct multi-step forecast → inverse-scaled output bands.

Chronos-2 is a time-series model, not a language model. Its vocab_size in the Hugging Face config is 2 — one slot for PAD, one for the learned REG separator. Everything else flows as continuous, scaled, patched float tensors. The forward pass below covers a single batch of grouped series; no autoregressive sampling is involved — the quantile head emits the full horizon in one shot.

No tokens, no sampling
Continuous in — quantiles out

Original Chronos quantized values into a discrete token vocabulary and ran T5 as an autoregressive language model. Chronos-2 keeps values continuous, replaces token sampling with a direct multi-step quantile head, and shrinks vocab_size to 2 — only PAD and REG. arXiv 2510.15821

REG separator
One learned token between ctx & future

A single learned REG embedding is inserted at the boundary between context-patch tokens and future-patch tokens. The report describes it both as a separator and as an attention sink — a parking spot for diffuse attention mass that does not need to land on any specific patch. arXiv 2510.15821

03 · Centerpiece

Two attention axes · what makes Chronos-2 universal

Chronos-2's differentiator is not "it is a transformer"; it is the two-axis attention pattern that lets it mix temporal context within a series and mix related series within a group.

Every encoder block runs two distinct self-attentions back to back. TimeSelfAttention mixes patches along the temporal axis of a single series and uses Rotary Position Embeddings to know where in time a patch sits. GroupSelfAttention mixes patches across series that share a group_id at the same patch index, with no positional embedding — series in a group have no canonical order. Together they cover the four group semantics in the report.

layer[0] · within one series
Time self-attention · RoPE

For each series independently, every patch attends to every other patch in the same series (context + REG + future). Rotary Position Embeddings are applied to Q and K with rope_theta = 10000, so the model knows the relative time offset between two patches.

layer[1] · across series sharing a group
Group self-attention · no positional embedding

At each patch index, the patch from series s attends to the patches at that same index from every series with the matching group_id. No RoPE, no positional embedding — series in a group are an unordered set. This is what enables multivariate forecasting, cross-learning, and covariate handling in the same architecture.

group semantic 1
Independent univariate

One series per group. Group attention degenerates to identity — each series is forecast in isolation. Equivalent to running Chronos-Bolt-style univariate forecasting in a batch.

group_ids = [0, 1, 2, …, N−1]
group semantic 2
Cross-learning / cold-start

Related items (same SKU family, same store region, shared metadata) put in one group. Information flows between siblings in-context — useful for short or new series.

group_ids = [0, 0, 0, 1, 1, …]
group semantic 3
Multivariate variates

Variates of one multivariate system — e.g. price, load, generation in one energy market. All variates share a group_id; the model jointly forecasts the multivariate vector.

group_ids = [0, 0, 0, …]
group semantic 4
Target + covariates

One forecasting task: target(s) plus past-only covariates plus known future covariates, all in one group. Covariate predictions are computed but the loss only flows through target dims.

group_ids = [0, 0, 0, 0]
✓

Why this matters

Every other large time-series foundation model in the listed sources is either strictly univariate or relies on external covariate regressors stacked on top. Chronos-2 lifts that distinction into the architecture: a single forward pass handles univariate, cross-learning, multivariate, and covariate-informed tasks — the difference is just how you set group_ids. arXiv 2510.15821 Amazon Science blog

05 · Input construction

Historical context, future covariates, and masks

Chronos-2 keeps historical and future inputs in separate tensors with paired masks so it can encode “target unknown, covariate known” cleanly.
x_hist · historical context
What the model sees from the past
  • Target(s)Observed target values for the last ≤ 8192 steps
  • Past covariatesReal and categorical regressors observed alongside the target
  • Known-future covsObserved values in the context window are also placed here
  • Observed maskBinary mask flags missing or padded history
  • Time indexi / context_length per step (Step 4)
x_future · future inputs
What the model is allowed to peek at
  • Target(s)Zeros + mask — future target is what we are predicting
  • Known-future covsCalendar features, planned promos, holidays, etc.
  • Past-only covsZeros + mask — not known into the future
  • Future-cov maskBinary mask flags which future cells are actually known
  • Time indexRuns from 0 forward, scaled by context length
Scaling
Robust instance norm

Per-series loc and scale are computed from observed history. Values are normalized before patching; an optional arcsinh-style variance stabilization can be enabled for heavy-tailed targets. Future covariates are normalized with the same per-series statistics so context and future live on the same scale.

Categorical covariates
Target vs ordinal encoding

Per the report, categorical covariates are target-encoded when the task is univariate, and fall back to ordinal encoding when the task is multivariate (no single target to encode against). arXiv 2510.15821 §3

Loss masking
Targets only

The model also produces forecasts for covariate dimensions, but those forecasts are ignored downstream. Loss is computed only against target dimensions, and missing target steps are masked out of the loss too.

06 · Patch & REG

From normalized values to 768-D patch tokens

Non-overlapping 16-step patches; concatenated time/value/mask channels; one learned REG separator at the context-future boundary.
Patch construction
16-step non-overlapping patches

Context values are truncated to the last 8192 steps before patching, then split into ≤ 512 non-overlapping 16-step patches. Each patch concatenates a 16-D time encoding, the 16 normalized values, and the 16 mask bits. The full 48-D vector is mapped to 768-D by a two-layer residual MLP.

REG token
Separator + attention sink
i

Future time encoding

Future time encoding runs from 0 forward and is scaled by context_length, so a forecast of horizon h with a context of L always lands in the same numeric range regardless of how long that particular context happens to be. Context loc/scale are reused for future covariate normalization — the model never sees raw target units. arXiv 2510.15821

07 · Output head

21-quantile direct multi-step forecast head

Chronos-2 emits the full prediction horizon in one shot. No autoregressive token sampling; no Monte-Carlo passes for uncertainty.
From future hidden states to quantiles
Output pipeline
  1. Slice hidden states for the future patches only: H_fut ∈ ℝ^(B × N_fut × 768).
  2. Apply a ResidualBlock that maps 768 → 21 · 16 values per future patch.
  3. Rearrange to [B, 21, horizon], where horizon = N_fut · 16.
  4. Inverse instance normalization: y = ŷ · scale + loc per series.
  5. Loss (training) or output (inference) over target dimensions only; covariate-dim forecasts are dropped.
# conceptual
H_fut   = hidden_states[:, :, n_ctx+1:, :]    # skip ctx + REG
flat    = res_block(H_fut)                     # [B, N_fut, 21·16]
q       = rearrange(flat, "b n (q t) -> b q (n t)")
y       = q * scale + loc                  # [B, 21, horizon]
21 quantile levels
From q=0.01 to q=0.99
0.01 0.05 0.10 0.15 0.20 0.25 0.30 0.35 0.40 0.45 0.50 0.55 0.60 0.65 0.70 0.75 0.80 0.85 0.90 0.95 0.99

The two extreme quantiles — 0.01 and 0.99 — are what differentiate Chronos-2's coverage at the tails: extreme-quantile output improves rare-event coverage and risk-aware forecasting, per the report.

08 · Training

Objective, schedule, and data mix

A two-stage context schedule, quantile-regression loss, real + synthetic time-series corpora. Exact compute and optimizer details are not reported in the listed sources.
Objective
Quantile regression loss

For each target step in the horizon and each of the 21 quantile levels τ, Chronos-2 minimizes the standard pinball loss ρ_τ(y − ŷ_τ). Known-future covariate values and missing target values are excluded; only observed target steps contribute to the loss.

# pinball / quantile loss
L = mean(
        max(τ·(y−ŷ_τ), (τ−1)·(y−ŷ_τ))
    ) # over target steps × 21 quantiles

Reported throughput
Over 300 forecasts/s on one A10G

Chronos-2 delivers more than 300 series-level forecasts per second on a single NVIDIA A10G GPU under the conditions defined by the report.

  • Batch size1024 series
  • Context length2048 steps
  • Prediction length64 steps
  • GPUNVIDIA A10G
Why it can be both bigger and fast
Direct multi-step amortization

Original Chronos sampled future values one token at a time; the new direct multi-step head produces all 21 quantile trajectories in a single forward pass. Combined with the encoder-only backbone — no decoder cross-attention, no per-step cache — throughput stays high even though the parameter count moved to 120 M.

Domain case study
Energy price forecasting

Day-ahead market price as the target, with load and renewable generation forecasts as known future covariates. The report uses this case to illustrate how a single Chronos-2 inference call ingests target history and future covariates jointly — no separate covariate regressor required.

arXiv 2510.15821 §7
Domain case study
Rossmann retail sales

Store-level sales with footfall, store-open / closed flags, ongoing promotions, and holiday calendars as covariates. Group attention lets siblings in a store cluster share information; future covariates land directly in the same forward pass.

13 · Caveats

What is and is not in the listed public sources

Explicit list of items reported, items inferred from code, and items not disclosed.
i

Encoder-only · not the original autoregressive T5

Chronos-2 is described and configured as a single encoder stack ("T5-inspired") with a direct multi-step quantile head. It is not the original Chronos discrete-token autoregressive setup — values are continuous, vocab_size = 2 only for PAD/REG, and no decoder cross-attention is used.

i

Covariate handling vs predecessors

Original Chronos and Chronos-Bolt can be combined with external covariate regressors at the application layer, but neither model natively conditions on covariates inside its own forward pass. Chronos-2 does. When the report contrasts "native" vs "external regressors" treat this as the meaning.

!

Norm placement & residual order

The exact placement of LayerNorm (pre vs post) and the residual ordering inside the encoder block should be confirmed against layers.py at the pinned commit. We have shown a standard T5-style pre-norm arrangement; if the source code differs, follow the source code.

!

Items not reported in listed sources

Exact pretraining corpus size, number of training patches/steps, compute budget, hardware, optimizer, learning-rate schedule, batch size, training duration, and any hidden implementation details beyond the public code/config are not in the listed sources. Where a column or label says not reported, we mean exactly that.
