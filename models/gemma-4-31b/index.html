Gemma 4 31B Dense

A 30.7 B-parameter dense decoder-only transformer built from Gemini 3 research, released April 2, 2026 under Apache 2.0. The 31B is the largest member of the Gemma 4 family and the only variant without an MoE option, targeting workstations and single consumer GPUs. The defining architectural choice is a hybrid attention pattern that interleaves local sliding-window layers with full global layers — with the final layer always global — paired with unified KV and proportional RoPE on the global layers to keep the long-context memory footprint manageable.

02 · End-to-end map

From multimodal input to a text logit vector.

A prompt arrives as an interleaved sequence of text tokens and image regions. Images are first lifted into the language-model token stream by a ~550 M vision encoder; the resulting joint sequence then flows through 60 decoder layers and exits as logits over the 262 K Gemma vocabulary. Audio is not supported on the 31B variant — only the E2B and E4B models embed an audio encoder.

03 · The 60-layer stack

Local sliding window, interleaved with global, always global at the top.

The Gemma 4 model card states the family interleaves local sliding-window attention (1 024-token window on 31B) with full global attention, and that the final layer is always global. The 31B has 60 layers total. The model card does not publish the exact interleave ratio for the 31B; the ribbon below uses Gemma 3's published 5 : 1 local : global pattern as a visual placeholder — it is marked accordingly and should not be read as a confirmed Gemma 4 layout.

04 · Decoder block internals

Pre-norm transformer block with GQA + QK-Norm + dense MLP.

The Gemma family's published block uses a pre-norm arrangement, RMSNorm, grouped-query attention with QK-Norm, and a SwiGLU-gated dense MLP. The Gemma 4 model card text we fetched does not re-state head counts, hidden size, or MLP intermediate size for the 31B, so the diagram below shows the structure verified from the inventory's Raschka entry — Dense Transformer · GQA · QK-Norm · local-global attention — and marks numeric shapes as not reported where the model card is silent.

1 · Pre-attention RMSNorm
RMSNorm

Pre-norm transformer arrangement: normalize the residual stream before entering the attention sub-block. RMSNorm uses learned scale γ, no bias.

2 · Grouped-query attention
GQA · inventory lead

Multiple query heads share each KV head. The Gemma 4 31B head and group counts are not stated in the fetched model card; GQA itself is the Raschka-table attribute carried forward from Gemma 3.

3 · QK-Norm on Q and K
per-head RMSNorm

An additional RMSNorm applied to the per-head query and key vectors before the dot product. Stabilises attention logits at long context; carried over from Gemma family per inventory lead.

4 · Local or global attention
window 1 024 · or · full

Layer type is fixed at config time. Local layers attend only within the most recent 1 024 tokens; global layers attend over the entire 256 K window and additionally apply p-RoPE and unified KV.

5 · Residual add
x + Attn(x)

Standard residual connection around the attention sub-block.

6 · SwiGLU MLP + residual
gate · up · down

Gated MLP: W_down( SiLU(W_gate·h) ⊙ (W_up·h) ). Standard Gemma-family channel mixer; intermediate width not stated in the fetched model card.

05 · Local vs global attention

Two attention regimes alternating along the stack.

The Gemma 4 model card describes the design as: "a hybrid attention mechanism that interleaves local sliding window attention with full global attention, ensuring the final layer is always global. This hybrid design delivers the processing speed and low memory footprint of a lightweight model without sacrificing the deep awareness required for complex, long-context tasks." The two regimes share QKV projection geometry but differ in mask, positional encoding, and KV caching.

Local sliding-window attention

Each query attends only over the most recent 1 024 tokens of context. Cheap, constant-memory per token, regardless of how long the full prompt is.

06 · p-RoPE & unified KV

Two memory tricks that ride on global layers.

The Gemma 4 model card calls out two long-context optimisations applied only to the global layers: Proportional RoPE (p-RoPE) for positional encoding, and unified Keys and Values in the KV cache. Local layers do not carry these. The model card text introduces the names without a numerical recipe, so this section presents the behaviour the names imply and labels the formal definitions as not reported.

Long-context memory · global layers only

Proportional RoPE + shared K / V

p-RoPE. The Gemma 4 model card names Proportional RoPE as the positional encoding used on global layers. The precise frequency-scaling recipe is not stated in the fetched text, but the model-card phrasing — "To optimize memory for long contexts, global layers feature unified Keys and Values, and apply Proportional RoPE (p-RoPE)" — pairs p-RoPE explicitly with long-context behaviour, suggesting per-dimension base frequencies scaled with context length.

Unified Keys and Values. Global layers tie the K and V projections so they share storage in the KV cache, roughly halving the per-token cache footprint at those layers. For a 256 K window across 60 layers, that saving compounds.

Why only on global layers? Global attention is the bottleneck at long context — its KV cache grows with the full sequence. Local layers cap at the 1 024-token window, so the cache there is already bounded; the optimisations target the layers where memory actually hurts.

Global-layer attention · sketch
K = V = W_KV · h // unified projection
Q = W_Q · h
Q', K' = p-RoPE(Q, K, pos) // proportional rotation
A = softmax( Q' · K'ᵀ / √d_h ) · V // causal · full context
Exact head dim, frequency base, and projection shapes — not reported in the fetched Gemma 4 model-card text.
07 · Vision tower

Variable-resolution image input as a token budget.

The Gemma 4 31B carries a ~550 M-parameter vision encoder that turns images and video frames into a sequence of visual tokens for the language model. The number of tokens per image is configurable — a budget of 70, 140, 280, 560, or 1 120 — letting callers trade detail for throughput. Multimodal prompts should place image content before text. Audio is not supported on the 31B.

08 · Thinking mode

A built-in reasoning channel, toggleable per prompt.

Gemma 4 31B is shipped with a configurable thinking mode: include a <|think|> token at the start of the system prompt and the model writes its internal reasoning into a separate channel before producing the final answer. The 31B always emits the channel tags — even when thinking is disabled — but leaves the thought block empty in that case. Thought content must never be carried into subsequent turns.

11 · Caveats & unknowns

What the fetched model card does not say.

The Gemma 4 model card is dense at the family / capability level but sparse at the per-shape level. Several quantities a deployer or researcher would want for Gemma 4 31B specifically are not in the fetched text and have been left as not reported throughout this page.

Hidden size and head geometry

The Gemma 4 model card does not state d_model, n_heads, n_kv_heads, or head_dim for the 31B. The HF config file is the canonical source for these — confirm there before quoting numbers in downstream material.

MLP intermediate width

The SwiGLU intermediate dimension is not in the fetched model-card text. Confirm in config.json on the HF repo.

Local / global interleave ratio

The model card states the family interleaves local and global attention and that the final layer is always global, but does not publish the per-layer pattern for the 31B. The ribbon in §3 uses Gemma 3's 5 L : 1 G pattern as a placeholder only.

Positional encoding on local layers

p-RoPE is specified for global layers. The positional scheme used on local layers is not separately described; whether it is also p-RoPE, vanilla RoPE, or implicit-via-window is unclear from the fetched text.

GQA & QK-Norm

GQA and QK-Norm appear in the inventory's Raschka-table entry for Gemma 4 31B; they are also Gemma-family conventions. They are not re-stated by name in the fetched 2026-04-17 model-card text. Treat as inventory-led, not model-card-confirmed.

Training token budget & recipe

The model card describes the data mix and cutoff (Jan 2025) but not the token count, optimiser, learning-rate schedule, post-training stages, or distillation recipe. Compute and energy figures are also absent from the fetched text.

Per-Layer Embeddings (PLE)

PLE is described on the model card as an E2B / E4B feature for on-device efficiency. The 31B is the dense workstation variant and does not use PLE in the model-card text.

Quantisation & serving

The HF repository advertises BF16 tensor type for the base release; the 31B has 155 community quantised mirrors. None of those are first-party. Quantisation-aware-training details, if any, are not in the model-card text.
