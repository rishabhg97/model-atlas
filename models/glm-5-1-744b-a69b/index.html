Release · 2026-04-07 · Z.ai (formerly Zhipu AI · THUDM)

A 744 B-parameter MoE with DeepSeek Sparse Attention.

GLM-5.1 is the open-weight, MIT-licensed flagship from Z.ai (formerly Zhipu AI), released as a post-training upgrade to GLM-5. Architecture is unchanged from GLM-5: a sparse Mixture-of-Experts decoder-only Transformer, 744 B total parameters and 40 B active per token, built around the DeepSeek Sparse Attention (DSA) module — an MLA-based attention with a Lightning Indexer that sparsifies the long-context path. A Multi-Token Prediction (MTP) head is shipped for speculative decoding. The 5.1 release targets long-horizon agentic coding through a revised RL pipeline (Slime), not new pretraining compute.

01 · Forward pass

From tokens to speculative draft.

GLM-5.1 is a pure text decoder — no native vision or audio tower in this release. Tokens enter a BPE embedding, traverse 78 decoder layers (4 dense + 74 MoE) where each layer's attention is either full MLA or sparse DSA per a fixed per-layer pattern, then exit through the LM head and an MTP NEXTN head that drafts up to 3 future tokens for speculative decoding.

02 · Layer stack

78 layers, a per-layer F/S sparse-attention pattern.

The shipping config.json exposes 78 decoder layers and an index_topk_pattern string of length 78: each character is F (full MLA attention) or S (sparse DSA attention with Lightning Indexer). Layers 0–3 are dense MLPs — the warm-up before routing begins; layers 4–77 are MoE with 256 experts and top-8 routing per token. The F/S choice is baked into the checkpoint and is not learned at inference time.

03 · Decoder block

Pre-norm residual block · attention path is the only switch.

Every block follows the standard pre-norm pattern: RMSNorm → attention → residual → RMSNorm → feed-forward → residual. The only variable is which attention module sits in the upper arm: full MLA on F layers, DSA with Lightning Indexer on S layers. Below: the two block variants drawn side-by-side.

04 · Attention · F layers

Multi-head Latent Attention — the dense backbone.

On F layers, GLM-5.1 runs the same MLA module DeepSeek introduced with V2/V3: keys and values are compressed into a low-rank latent c_t^KV per token, then up-projected per head at attention time. Position is carried by a small decoupled RoPE branch. The per-token KV cache is dominated by c_t^KV (independent of head count), which is what makes 200K context affordable.

Attention module · F-pattern layers only

MLA · low-rank KV compression + decoupled RoPE

all 200K context W_O → residual stream

MLA is the same module used in DeepSeek-V3 and Kimi K2: cache the low-rank c_t^KV per token, absorb the up-projections W_UK and W_UV into the query path at inference, and carry RoPE on a small dedicated subset of the head dimension so the absorption trick stays valid.

On F-pattern layers GLM-5.1 attends over the full 200K context window — this is what the Lightning Indexer on S-layers shortcuts past. The two attention modes share the same MLA weights; only the key-selection step differs.

05 · Attention · S layers

DeepSeek Sparse Attention — the cost-control surface.

DSA is the same mechanism Z.ai describes on GitHub: a Lightning Indexer scores every past key cheaply, the top-k are selected, and the heavyweight MLA softmax runs only over those selected tokens. The result is sub-quadratic attention with a learned key-selection policy — the published GLM-5 launch positions it as "largely reducing deployment cost while preserving long-context capacity." The per-layer top-k parameter is part of the config (the index_topk_pattern); the exact k values per layer are not enumerated in the public model card.

Attention module · S-pattern layers only

DSA · Lightning Indexer + top-k key selection

selected keys) o_t = softmax(Q_t K_𝒯ᵀ / √d_h) · V_𝒯 · W_O → residual Two-stage attention: Indexer is cheap (over all N), MLA softmax is heavy but runs only over the selected k. Net: O(N) indexer + O(k · d_h) attention per query — sub-quadratic in N.

DSA's key contribution is decoupling "which tokens matter" from "compute attention". The Lightning Indexer is a small scoring head that runs over the entire history at low cost; the MLA softmax then only operates on the top-k keys it returns. This is the trick that lets GLM-5.1 hold 200K context windows at deployment cost similar to a much smaller dense model.

The position of S-layers in the stack is not uniform: the index_topk_pattern string interleaves them in a fixed, presumably trained pattern. The first three layers are all F (paired with dense MLPs); sparse layers cluster in the middle and tail.

06 · Feed-forward

256 routed experts, top-8 per token, plus shared.

Layers 4–77 replace the dense MLP with a routed Mixture-of-Experts: 256 experts per MoE layer, top-8 selection, plus one or more shared experts that run on every token regardless of routing (their existence is confirmed by the NVFP4 quant card, which excludes them from quantization; the exact count is not enumerated in the public model card). The 40 B active-per-token figure is what falls out of routing the dense path through 8 of 256 experts.

07 · MTP head

Multi-Token Prediction — speculative drafting in the model.

GLM-5.1 ships an auxiliary MTP head (Multi-Token Prediction, also called NEXTN) attached to the main trunk. At inference, the head drafts up to 3 future tokens in a single forward pass; the verifier accepts them via probabilistic rejection sampling. Community deployments wire it through vLLM's speculative_config with method: "mtp", num_speculative_tokens: 3. MTP cuts latency at the cost of a small accept-rate-dependent compute overhead per token.

Speculative decoding loop (per accepted prefix length ℓ ∈ {1..4})
1. Trunk forward at position t → h_t
2. LM head: y_{t+1} ~ p(· | h_t)  # verifier token
3. MTP head(h_t) → ŷ_{t+2}, ŷ_{t+3}, ŷ_{t+4}  # 3 drafts in one shot
4. Verify drafts against trunk; accept longest prefix where draft prob aligns
5. Resume at t + ℓ
vLLM config (per NVFP4 quant card): method=mtp · num_speculative_tokens=3 · rejection_sample_method=probabilistic
08 · Pretraining

GLM-5 base, 28.5 T tokens — 5.1 inherits.

GLM-5.1 is explicitly a post-training upgrade: the architecture, vocabulary, embeddings, and pretrained checkpoint are inherited from GLM-5 (February 11, 2026). GLM-5's launch describes scaling from GLM-4.5's 355B/32B-active to 744B/40B-active and growing pretraining data from 23T → 28.5T tokens. The release adds DSA as a deployment-cost optimization that was not present in the GLM-4.5 line.

09 · Post-training

Slime — the real GLM-5.1 release.

Z.ai's repo (github.com/zai-org/GLM-5) names Slime as a "novel asynchronous RL infrastructure that substantially improves training throughput and efficiency, enabling more fine-grained post-training iterations." The framework is open-sourced separately at THUDM/slime. GLM-5.1's delta over GLM-5 is described as predominantly an RL pipeline revision targeting long-horizon coding, agentic tool use, and autonomous execution; not additional pretraining.

What 5.1 changes vs 5
all per Z.ai's release notes and the GLM-5.1 model card; specific recipes not disclosed
  • Long-horizon coding. Z.ai demonstrated GLM-5.1 running an 8-hour autonomous coding session (655 iterations, building a Linux desktop environment from scratch).
  • Agentic tool use. Tool-call parser identified as glm47; reasoning parser as glm45 (per community deployment configs).
  • RL throughput. Slime's async design lets Z.ai run more post-training cycles per unit of compute — the stated driver of 5.1's gains over 5.
  • Same weights at the base. Architecture identical to GLM-5; the 5.1 checkpoint differs only in post-training-modified parameters.
10 · Hardware

Trained on 100,000 Huawei Ascend 910B — no NVIDIA GPUs.

Reporting consistent across multiple outlets places the GLM-5 / GLM-5.1 training cluster on Huawei Ascend 910B chips (HiSilicon design, SMIC 7nm) running the MindSpore framework — a fully domestic Chinese stack. Z.ai has been on the US Entity List since January 2025, ruling out NVIDIA, AMD, and Intel accelerators for their training runs. The deployment side is hardware-agnostic and runs on NVIDIA inference engines (vLLM, SGLang) via standard FP8/BF16 paths.

12 · Caveats

What the public sources do not pin down.

The GLM-5.1 release is unusually opaque about architectural micro-detail compared to the DeepSeek-V3 or Kimi K2 papers it lineages from. The model card lists the high-level topology and ships the weights, but several dimensions that would normally appear in a foundation-model brief are absent. Below is an explicit ledger.

Active-parameter count: 40 B vs "A69B"

The atlas inventory titles this model "GLM-5.1 744B-A69B." No primary source we located uses 69 B. The official GitHub README, the Hugging Face model card, llm-stats.com, Unsloth's deploy guide, and the community NVFP4 quant card all say 40 B active. We use 40 B everywhere in this page.

Hidden size · attention heads · per-head dim

d_model, head count, and d_h are not enumerated in the published Z.ai materials. The community NVFP4 quant card reads them from the checkpoint config but does not republish them. Unknown in source inventory.

Vocabulary size & tokenizer

Tokenizer identity, BPE corpus, and vocab size are not stated on the GLM-5.1 model card or in Z.ai's release blog. Unknown in source inventory.

DSA top-k values per layer

The 78-character F/S pattern is published but the actual k selected by the Lightning Indexer at each S-layer is not. Top-k presumably varies per layer (the parameter exists in the config and is supplied as an env override), but the values are not enumerated. Not reported.

Shared-expert count

The NVFP4 card confirms shared experts exist (they are excluded from quantization) but does not name how many. Configurations from the DeepSeek-V3 family use one shared expert; GLM-5.1 may differ. Count not reported.

Router scoring & load balancing

The model card lists n_experts = 256 and top-k = 8 but does not specify the router affinity function (softmax vs sigmoid), the auxiliary load-balance loss, or any bias-routing scheme. Unknown in source inventory.

Optimizer · LR schedule · training compute

The 28.5 T pretraining token count is from Z.ai's GitHub README. Optimizer, batch size, learning rate schedule, and total training-step count for the GLM-5 base or the 5.1 post-training delta are not disclosed for either model.

Slime RL recipe

Slime is open-sourced as infrastructure, but the GLM-5.1 RL recipe — reward model, algorithm (PPO / GRPO / DAPO / other), KL constraints, prompt mix — is not published. Recipe not disclosed.

Modalities

GLM-5.1 is a text-only decoder; no vision, audio, or video tower in this release. The chat.z.ai product surfaces images, but as a separate tool layer above the model. Confirmed by model card.

Vendor benchmark methodology

SWE-Bench Pro 58.4 / coding composite 54.9 / HLE 50.4 / BrowseComp 75.9 are all Z.ai-published figures, some inherited from GLM-5. Competitor scores were not all re-run under Z.ai's harness; treat all bars as vendor-comparable, not independent third-party measurements.
