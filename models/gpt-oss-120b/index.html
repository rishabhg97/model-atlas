OPEN-WEIGHTSparse MoE Transformer · Reasoning

Sparse MoE Transformer with alternating sliding/full attention

OpenAI’s first open-weight release since GPT-2. A 36-layer decoder-only Transformer where every dense FFN slot is replaced with a 128-expert MoE, and attention layers alternate between a tight 128-token sliding window and full causal attention. Trained for chain-of-thought reasoning and ships with the “Harmony” chat format.

02 · Layer stack

36 blocks, two alternating attention modes

The config.json layer_types array enumerates an exact sliding, full, sliding, full… sequence across all 36 layers. Every layer has the same MoE FFN beneath it; only the attention head varies.

Layer index → (attention kind, MoE FFN)

Sliding-window attn · w=128 Full causal attn MoE FFN · 128 experts, top-4
Sliding-window attention
18 layers odd indices 1, 3, 5, … 35
Each query attends only to the most recent 128 keys. Tiny KV-cache footprint dominates the backbone.
Full causal attention
18 layers even indices 2, 4, 6, … 36
Global, causal attention over the entire 128K context. Provides the cross-context lookup the sliding layers can't.
03 · Block internals

Pre-norm decoder block with residual fan-out

Each of the 36 blocks is two pre-norm residual sub-layers: an attention sub-layer (sliding or full, depending on parity) and an MoE feed-forward sub-layer. RMSNorm is applied before each sub-layer; linear biases are retained on attention projections (attention_bias=true — uncommon for modern LLMs).

Block forward

x ∈ ℝ^[B, T, 2880]  →  x' ∈ ℝ^[B, T, 2880]

+ Residual add  →  h = x + attn(norm(x)) [B,T,2880]

04 · Attention

Grouped-Query Attention, two ranges

All 36 layers use the same projection topology: 64 query heads → 8 key/value heads (a GQA group size of 8) with head_dim = 64. What varies layer-to-layer is the mask: half the layers are limited to a 128-token sliding window; the other half are full causal.

SLIDING
Sliding-window attention

Each query token attends only to itself and the previous 127 keys. With w = 128, KV-cache for these 18 layers is bounded — total KV state across the sliding stack is O(T·8·64·18) for a window of size T, but only 128 entries are looked at per query.

05 · Positional encoding

RoPE with YaRN scaling from 4K → 128K

Positions enter through Rotary Position Embeddings on Q and K with base θ = 150 000. The original training window of 4 096 tokens is stretched to 131 072 via YaRN with factor = 32 — applied per-frequency band, not as a global rescale.

06 · Mixture-of-Experts

128 experts per layer · top-4 routing

Every block’s feed-forward sub-layer is a sparse MoE. For each token a linear router scores all 128 experts; the top-4 are softmax-normalized and their SwiGLU MLP outputs are linearly combined. There is no shared expert and no expert-bias term in the released config — pure top-k softmax routing.

07 · Tokenizer · Harmony

o200k_harmony — 201 088 BPE tokens, role-aware

GPT-OSS ships its own BPE tokenizer derived from o200k. The Harmony chat format wraps system / developer / user / assistant turns with explicit role tokens and supports multi-channel assistant outputs (analysis vs. final).

08 · Quantization

MoE weights in MXFP4 · attention & embeddings in BF16

The released checkpoint quantizes only the expert MLP weights to MXFP4 (microscaled FP4 with shared exponents per 32-element block). Attention projections, the router, embedding table, and LM head stay in BF16 / FP16. This is what makes the 116.8 B-param model fit in roughly 80 GB of VRAM.

10 · Reasoning effort

Three configurable thinking budgets

A controllable property of the released model is its reasoning budget, surfaced as a system-prompt setting. Higher levels produce longer chain-of-thought traces in the analysis channel before the final answer.

REASONING: LOW
Short answers, minimal CoT

Used for quick replies, formatting tasks, and high-throughput agentic loops where latency dominates.

analysis channel ≈ terse
REASONING: MEDIUM
Balanced

Default for chat-style use. Reasoning traces are present but not exhaustive.

analysis channel ≈ moderate
REASONING: HIGH
Deliberative

Long, exhaustive chains for math, code, and multi-step agentic tool-use. Token usage is the trade-off.

analysis channel ≈ long
