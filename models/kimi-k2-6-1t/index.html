A trillion-parameter MoE that behaves like 32B per token.

Kimi K2.6 is a native multimodal agentic model from Moonshot AI: a 1-trillion-parameter sparse Mixture-of-Experts decoder built on Multi-head Latent Attention (MLA), with a 400M-parameter MoonViT vision tower attached for image and video. The model card explicitly states "Kimi-K2.6 has the same architecture as Kimi-K2.5" — the topology is fixed, and the K2.6 release lives in the post-training recipe: long-horizon coding, swarm coordination, and a hallucination-rate drop from 65 % to 39 % on AA-Omniscience.

01 · Forward pass

From mixed-modality input to next-token decision.

Text and visual tokens funnel into a single embedding stream, traverse 61 decoder layers (one dense, sixty MoE), and exit through a tied LM head. The model is decoder-only and autoregressive; the only modality-specific surface is the MoonViT tower that converts pixels and video frames into token embeddings before they enter the trunk.

02 · Layer stack

One dense, sixty mixture-of-experts.

The published configuration on build.nvidia.com reports 61 layers, of which one is a dense MLP; the remaining sixty are MoE layers, each with 384 routed experts plus a single shared expert. The dense layer is the warm-up — every subsequent block conditionally routes its feed-forward computation to 8 of 384 experts per token.

Why one dense layer? Kimi K2's design (inherited by K2.5 and K2.6) follows the DeepSeek-V3 convention: the first transformer layer uses a standard MLP, and only the higher layers introduce the routed MoE. The dense layer keeps token-level features dense before the routing distribution has stabilized.
03 · Decoder block

Pre-norm residual block, MLA + MoE.

Every K2.6 layer is a standard pre-norm transformer block with two residual additions: one across attention, one across the feed-forward. The attention path is always MLA; the feed-forward path is a dense SwiGLU MLP at layer 1 and a routed MoE everywhere else.

04 · Attention

Multi-head Latent Attention — a compressed KV cache.

MLA is the single attention variant used on every one of the 61 layers. Instead of storing full keys and values per head, MLA projects them into a low-rank latent vector c_t^KV once per token, then reconstitutes per-head K and V on demand at attention time. This is what makes a 256 K-token context affordable.

Attention module · same on every layer

MLA · 64 heads · d_attn = 7168

MLA's win is that the per-token cache only stores the low-rank latent c_t^KV (plus the small RoPE-carrying head subset), not the full per-head K and V. At inference, the up-projection matrices W_UK and W_UV are absorbed into the query projection, so reconstructing per-head keys and values costs no extra memory bandwidth.

Position is handled by a decoupled RoPE branch: a small extra slice of the head dimension that never goes through the compression path, and instead carries the RoPE-rotated frequencies directly. The main compressed path stays RoPE-free, which is what makes the absorption trick valid.

05 · Feed-forward

384 routed experts, 1 shared, top-8 per token.

The MoE feed-forward replaces a single MLP with 384 small SwiGLU experts plus one always-on shared expert. A learned router picks the 8 highest-scoring experts for each token, weights their outputs, and adds the shared expert's output on top.

06 · Vision tower

MoonViT — 400 M parameters, native to pretraining.

K2.6 accepts text, images, and video in the same context window. Visual input is processed by MoonViT, a 400 M-parameter vision encoder that emits visual tokens directly into the same embedding space as the text tokens — the trunk does not distinguish between them once they enter layer 1. K2.6 adds native video input (mp4, mov, avi, webm) on top of K2.5's image support.

07 · Reasoning mode

Two decode modes from one set of weights.

K2.6 exposes a single chat surface with two decode modes: Thinking, which emits a structured reasoning channel before the final answer, and Instant, which skips it. Mode is chosen at request time via the chat template; the model weights are the same.

08 · Orchestration

Agent Swarm — 300 sub-agents, 4,000 coordinated steps.

Agent Swarm is the orchestration layer that sits on top of the model. K2.6 acts as the coordinator; it decomposes a complex prompt into parallel domain-specialised sub-agents, dispatches them, and synthesises their outputs. The headline numbers — 300 sub-agents and 4,000 steps — are roughly a 3× scale-up over K2.5's 100 / 1,500 ceiling.

09 · Pretraining

MuonClip — Muon with per-head QK clipping.

The published Kimi K2 foundation paper describes MuonClip, an extension of the Muon optimizer introduced with a technique Moonshot calls QK-Clip: a per-head clipping rule applied to the query and key projections with threshold τ = 100. The K2 base was trained on 15.5 T tokens across four domains; K2.6's incremental training compute over K2.5 was not disclosed.

10 · Post-training

Where the K2.6 release actually lives.

The architecture is unchanged from K2.5. Every K2.6 gain — long-horizon coding, lower hallucination, 300-agent swarm coordination — is attributed to a revised post-training pipeline. Moonshot has not published the recipe; what is documented is the outcome: a hallucination drop from 65 % to 39 % on AA-Omniscience and a +6 / +16 step on SWE-Bench Pro / Terminal-Bench vs K2.5.

11 · Quantization

INT4 native, not post-hoc.

Unlike retro-fitted INT4 quantization that ships as a separate file, K2.6's weights are produced through quantization-aware training: the model is trained knowing it will be served at INT4. That cuts the on-disk footprint to roughly 594 GB and yields a ~2× generation speedup over the BF16 reference, with no measurable quality drop on reported benchmarks.

13 · Caveats

What the source inventory does not pin down.

K2.6's model card and the build.nvidia.com mirror agree on the structural numbers above — but several details that would appear in a full architecture brief are absent from the public sources at release time. Below is an explicit ledger of what is not confirmed and what is borrowed from sibling releases (K2 / K2.5).

Positional encoding rank

The model card does not state the exact RoPE rank used in the decoupled branch. MLA convention is a small d_r per head; the absolute value for K2.6 is unknown in source inventory.

Router scoring & load balancing

The K2.6 model card lists n_experts = 384, top-k = 8, and shared = 1, but does not specify the router's affinity function (sigmoid vs softplus vs softmax) or the load-balancing loss. Unknown in source inventory.

K2.6 training delta

Moonshot states the architecture is identical to K2.5 and that "more training compute" was applied, but does not disclose the K2.6-specific token budget, data mix, or RL recipe. Not reported.

Per-head dimension

Implied as d_attn / heads = 7168 / 64 = 112, but in MLA the practical per-head dim has two components (compressed + RoPE). The exact split is not given on the card.

Tokenizer identity

Vocab size is reported as 160 K. The model card states the tokenizer is shared with the K2 family but does not name a base BPE corpus. Tokenizer corpus not reported.

Vendor benchmark methodology

All scores in §12 are Moonshot-published. Some peer scores were re-run by Moonshot under their own harness (marked with asterisks on the model card); they should be treated as vendor-comparable, not independent third-party measurements.
