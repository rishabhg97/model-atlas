Overview

Liquid AI · Vision-language model family

Liquid LFM2-VL And LFM2.5-VL Architecture

A family of low-latency, on-device vision-language models built on Liquid's hybrid LFM2/LFM2.5 backbones, paired with a SigLIP2 NaFlex visual encoder. Image+text in → text out, with optional bounding-box grounding and text-only function calling.

What it does

Takes one or more images plus a text prompt (ChatML-style chat) and emits text. The text can be a free-form answer, a JSON array of object detections, or a tool call.

Where it runs

Built for on-device inference — phones, laptops, vehicles, browsers. Native bf16, plus official GGUF (llama.cpp) and ONNX checkpoints for the 450M tier.

What it doesn't do

Liquid explicitly flags the 450M as not suited for knowledge-intensive tasks or fine-grained OCR. No video, no audio, no image generation.

04

Image preprocessing & tiling

Aspect ratio is preserved without distortion. Images above the native cap are split into 512×512 non-overlapping patches. Thumbnail context is enabled for LFM2-VL-1.6B/3B and LFM2.5-VL checkpoints; the first-generation LFM2-VL-450M config sets use_thumbnail=false.

Native handling (≤ 512×512)

Images at or below 512×512 are processed at native resolution, without upscaling. Non-standard aspect ratios are preserved.

Max native512 × 512 px Upscalingnone Letterboxingno — aspect preserved Image tokens / image32 – 256 recipe; config default min 64

Tiling (> 512×512)

Large images are split into a grid of 512×512 non-overlapping patches. Where use_thumbnail=true, a single downscaled thumbnail of the full image is also encoded and prepended for global context.

05

Vision encoder — SigLIP2 NaFlex 86M

The visual front-end of LFM2.5-VL-450M is a shape-optimized SigLIP2 NaFlex encoder (~86M parameters). Its Hugging Face config reports a 12-layer, 768-hidden SigLIP2 vision tower with 16-pixel patches and a 256-patch grid before VL downsampling.

What it is

SigLIP2 is the second-generation Sigmoid-Loss for Image-text Pre-training encoder from Google. NaFlex is its variable-resolution, variable-aspect-ratio variant.

SigLIP2 NaFlex ~86 M params shape-optimized

Config details (450M)

Patch size16 px Hidden size768 Layers12 Attention heads12 Intermediate3,072 Headdisabled (vision_use_head=false)

06

Projector & multimodal fusion

After SigLIP2 emits its feature grid, a multimodal projector maps visual features into the LM token stream at the <image> tag. Liquid's LFM2-VL blog identifies the connector as a 2-layer MLP with pixel unshuffle; the LFM2.5-VL-450M config reports a GELU projector with hidden size 2048 and no projector layer norm.

What the sources do say

  • Each image consumes 32–256 image tokens, with the cap set at request time.
  • The released LFM2.5-VL-450M config default is min_image_tokens=64, while the model card's recommended recipe shows min_image_tokens=32.
  • Image tokens are inserted at the <image> placeholder inside a ChatML-style chat template.
  • The HF processor (AutoProcessor) handles the tile→token expansion automatically via apply_chat_template.
  • Multimodal fusion is decoder-only — there is no separate cross-attention head; image tokens enter the same residual stream as text tokens.

Config-backed projector facts

Connector2-layer MLP + pixel unshuffle Hidden activationgelu Hidden size2048 Biastrue LayerNormfalse Downsample2

Still not reported

  • Whether image tokens are pooled / compressed before insertion
  • How the 32–256 token budget is allocated across tiles and the thumbnail
Remaining caveat: the config exposes projector settings, but not a step-by-step tensor-shape trace from SigLIP2 features through pixel unshuffle into final LM embeddings.
07

LM backbone — LFM2 / LFM2.5 hybrid

The text-side backbone is an LFM2-family hybrid language model. Liquid's technical report describes LFM2 as a hybrid of gated short convolutions and grouped-query attention. For LFM2.5-VL-450M, the model card names the backbone as LFM2.5-350M and the released config exposes the concrete 16-layer topology.

Confirmed backbone facts (LFM2.5-VL-450M card)

450M config topology

The released LFM2.5-VL-450M config uses 16 backbone layers: 10 convolution layers and 6 full-attention layers, with 16 attention heads and 8 key/value heads where attention is used.

08

Output decoding

A single autoregressive head produces every output mode: free-form text, JSON-encoded bounding boxes for grounding, and tool-call tags for function calling. There are no separate task heads.

Token stream (illustrative)

<|startoftext|><|im_start|>systemYou are a helpful multimodal assistant by Liquid AI.<|im_end|> <|im_start|>user<image>·img··img··img··img·Describe this image.<|im_end|> <|im_start|>assistantThis image shows a C. elegans nematode.<|im_end|>

Reproduces the example chat template printed on the LFM2.5-VL-450M model card.

Sampling defaults

temperature0.1 min_p0.15 repetition_penalty1.05 min_image_tokens32 recipe · 64 config default max_image_tokens256 do_image_splittingTrue

Text parameters and the 32-token vision floor are from the 450M model card. The released config.json default uses min_image_tokens=64; both are inference-time knobs.

Free-form text

Captioning · VQA · multilingual chat · document description.

JSON-encoded boxes

For grounding queries the prompt asks for a JSON array of {label, bbox: [x1,y1,x2,y2]} with coordinates normalized to [0,1]. The boxes are emitted as ordinary text tokens — no separate detection head.

Tool-call tags

Function calling uses dedicated special tokens <|tool_call_start|>…<|tool_call_end|>. The chat template wraps the JSON tool schema for the model.

09

Grounding & task flows

Grounding and tool use are not separate modules — they are prompt patterns that ride the same autoregressive head. The model card shows two canonical patterns:

Visual grounding (RefCOCO-M style)

User asks for all instances of an object class; the model emits a JSON array of labeled boxes.

# prompt:
query = "statue"
prompt = f'Detect all instances of: {query}. Response must be a JSON array: '
         f'[{"label": ..., "bbox": [x1, y1, x2, y2]}, ...]. '
         f'Coordinates are normalized to [0,1].'

# response:
[{"label": "statue",
  "bbox": [0.30, 0.25, 0.40, 0.65]}]

Tool calling (text-only)

Tool schemas attach via apply_chat_template(..., tools=tools). The model emits a tagged call followed by user-facing acknowledgement.

# tool schema
tools = [{
  "name": "get_weather",
  "description": "Get current weather for a location",
  "parameters": {"type":"object",
                 "properties":{"location":{"type":"string"}},
                 "required":["location"]}
}]

# response:
<tool_call_start>[get_weather(location="Paris")]<tool_call_end>
I am retrieving the current weather for Paris.<|im_end|>

Grounding is benchmarked on RefCOCO-M (LFM2.5-VL-450M scores 81.28); tool calling is benchmarked on BFCLv4 (21.08). Both numbers come straight from the model card.

10

Latency & small-model design choices

Everything in the architecture leans toward on-device throughput: keep the encoder small, cap the image tokens per request, run a sub-1B-param hybrid LM, and expose the speed/quality dial as an inference-time parameter — no retraining required.

1. Compact visual front-end

86 M parameter SigLIP2 encoder. Roughly 20% of the 450M VL system's total parameter count — most of the budget stays in the LM.

2. Capped image tokens

A request never spends more than max_image_tokens on imagery (default 256). The 32-token floor is enough for compact thumbnails and small images.

3. Tunable tile count

Operators trim the tile grid (and the resulting visual token count) at inference, trading detail for latency without retraining.

4. Hybrid LM blocks

LFM2 / LFM2.5 use a mixed conv+attention architecture. Liquid markets this as the source of millisecond-class on-device latency on phones, laptops and vehicles.

5. Deployment-grade exports

Official GGUF (llama.cpp / CPU) and ONNX (cross-platform, mobile, edge) builds ship alongside the BF16 reference, plus a WebGPU browser demo for the 450M.

6. Domain scoping

Liquid recommends the 450M for general captioning, VQA and detection — and explicitly against fine-grained OCR or knowledge-intensive Q&A.

Training, data & post-training

Liquid now reports several VL-level training facts. First-generation LFM2-VL uses joint vision-language mid-training followed by joint SFT; LFM2.5-VL-450M scales the underlying pretraining from 10T to 28T tokens and adds preference optimization plus reinforcement learning focused on grounding, instruction following and reliability.

LFM2.5-VL-450M refresh (Liquid blog)

  • Scaled pretraining relative to LFM2-VL-450M: 10T → 28T tokens.
  • Post-training focused on production multimodal behavior.
  • Preference optimization and reinforcement learning were used to improve grounding, instruction following and reliability.
  • The LFM2 technical report frames the underlying LFM2 recipe as hardware-in-the-loop architecture search plus SFT, preference optimization and model merging.
Liquid reports the 10T → 28T scale-up for the LFM2.5-VL-450M refresh, but does not publish a separate image-pair count for that refresh.

LFM2-VL base training (Liquid LFM2-VL blog)

  • Built on LFM2 base models, then fused vision and language in a joint mid-training phase.
  • The text:image data ratio is gradually adjusted from 95% to 30%.
  • Followed by joint supervised fine-tuning with emphasis on image understanding.
  • Vision data combines large-scale open-source datasets and in-house synthetic vision datasets.
  • Overall LFM2-VL training is reported at order of 100B multimodal tokens.

Training pipeline — still not reported in listed sources

  • Image-text pairs count
  • Caption / VQA / grounding split
  • Named open-source dataset mixture
  • Supervised fine-tuning dataset names
  • Exact preference/RL algorithm for LFM2.5-VL
  • Distillation teacher(s)
  • Safety filtering pipeline
  • Whether SigLIP2 weights are frozen or co-trained
  • Resolution curriculum during VL training

User-facing fine-tuning is supported via Unsloth and TRL Colab notebooks linked from the model card — these are downstream SFT recipes, not training details for the base release.

Cross-source consistency

  • LFM2.5-VL family on the Liquid AI models page lists two 2.5-VL checkpoints (1.6B and 450M) — the workspace backlog row also lists 1.6B and 450M for LFM2.5-VL. Consistent.
  • Older LFM2-VL line in Liquid docs now lists 450M, 1.6B and 3B. The original August 2025 blog announced 450M and 1.6B; the October 2025 blog added 3B.
  • The Hugging Face card states the LFM2.5-VL-450M model size as 0.4B parameters (BF16). The product page label "450M" rounds up the combined LM (~350M) + vision encoder (~86M) total. Consistent.
  • The LFM2.5-VL-450M card recommends min_image_tokens=32, while the released config defaults to min_image_tokens=64. This page treats the card value as a recipe and the config value as the checkpoint default.
