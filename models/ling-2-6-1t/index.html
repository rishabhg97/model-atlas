01 · Forward pass

End-to-end signal flow

A token id enters the embedding layer and exits as a logit vector at the LM head. Most layers route their hidden state through a linear-state attention path; a small fraction route through a softmax MLA path. An auxiliary MTP head, attached after the main stack, predicts the next several tokens in parallel to enable speculative decoding at serve time.

02 · Hybrid attention stack

1 MLA layer for every 7 Lightning-Linear layers

Where Ling 2.0 used GQA throughout, Ling 2.5 / 2.6 swap most attention layers for a linear-state variant ("Lightning Linear") and keep a sparse anchor of full softmax MLA layers. The Ling-2.6-flash release post enumerates the exact ratio as 1 : 7, achieved through incremental training from the Ling 2.0 base. The same hybrid pattern carries over to the 1 T variant.

One 8-layer repeat

Each tile is one transformer block. Pattern repeats N / 8 times across the backbone. The position of the MLA layer within each group of 8 is not specified in the published cards — shown here as the last layer of each octet, a common convention.

03 · Decoder block

The two block shapes inside the backbone

Every block uses pre-RMSNorm with two residual sub-layers: an attention path (one of two variants) and a sparse-MoE FFN. The residual stream width and FFN configuration are shared across both block shapes; only the attention module differs.

04 · MLA attention

Multi-head Latent Attention — the global path

The minority block shape uses Multi-head Latent Attention: queries, keys, and values are projected into a low-rank latent that is what actually gets cached. At read time the latent is decompressed back to a per-head key/value pair. The cache footprint is much smaller than a vanilla MHA cache while preserving global softmax behavior on the residual stream.

05 · Lightning Linear attention

The majority path — sub-quadratic state recurrence

Seven of every eight blocks replace softmax attention with a linear-state attention module ("Lightning Linear" in the Ling-2.6-flash release post). Instead of scoring queries against a growing key/value cache, each layer carries a small fixed-size state tensor that is updated additively at each step. At serve time the state is allocated through SGLang's Mamba-style cache pool (--mamba-scheduler-strategy extra_buffer, --mamba-full-memory-ratio 1.4).

State recurrence at decode

O(1) per token
During autoregressive decoding the layer holds one matrix-valued state S. New tokens contribute an outer-product update; the read is a linear projection of S by the current query.

Why the hybrid wins for agents

design intent
Stated rationale from the Ling-2.6-flash release post and Ling-2.6-1T model card. All claims are vendor-reported.
Long-input throughput
Agent workflows feed long tool-trace and code histories. Linear-state cache holds size constant, so VRAM does not bloat with sequence length.
Latency on long output
Linear decode cost per token; the Ling-2.6-flash post reports up to ~4× higher decode throughput vs. comparable dense-attention models.
Where MLA is still needed
Pure linear stacks lose accuracy on long-range copy / recall (MRCR-style tasks). A 1 : 7 sprinkling of MLA anchors restores global routing without the full quadratic-cost tax.
06 · Sparse Mixture-of-Experts

Highly sparse routed FFN — ≈ 1/32 activation

Every FFN slot in the backbone is a sparse MoE bank. The Ling-2.0 release post for the predecessor Ling-1T model reports a 1/32 activation ratio — ~50 B active parameters out of 1 T total. The Ling-2.6-1T model card does not republish those numbers and does not enumerate routed-expert counts, top-k, or shared-expert count, but the configuration class (bailing_moe_v2_5) is a continuation of the same family.

One routing event (illustrative)

Each tile is one of an unspecified number of routed experts. For the current token, the router selects a small top-k; those experts fire, weighted by gate scores. One or more shared experts fire for every token.

FP8 mixed precision
Ling-1T was trained end-to-end in FP8 mixed precision — "the largest FP8-trained foundation model known to date," per the predecessor release post. The 2.6-1T safetensors continue to expose F8_E4M3, BF16, and F32 tensor types.
Heterogeneous pipeline
The Ling 2.0 release post lists per-layer types Embedding · Dense · MoE · MTP · Loss and reports an interleaved 1F1B pipeline customized for this heterogeneity — > 40 % end-to-end speedup over a homogeneous pipeline at trillion scale.
07 · Multi-Token Prediction (MTP)

Speculative decoding head trained alongside the LM

Ling 2.0 architecture diagrams include an explicit MTP layer appended to the backbone. At serve time, Ling-2.6-1T uses MTP through SGLang's EAGLE-style speculative-decoding driver: the MTP head proposes several next-token candidates from a single forward pass, and the main LM head verifies them — accepted candidates skip ahead, rejected ones are recomputed.

How MTP is wired into the forward pass

The MTP head is a small additional module that reads the final hidden state and produces logits for tokens beyond t+1. Both the LM head and the MTP head are trained jointly during pretraining (Ling 2.0 release post). At inference the MTP head runs as the speculative draft model; the main LM head is the verifier.

08 · Long context

256 K-token context window served by default

The SGLang inference recipe published with the model card launches with --context-length 262144 — a 256 K-token window. The bulk of that window is processed by the linear-state attention path; the 1-in-8 MLA layers retain global softmax on a much smaller fraction of the compute. Benchmarks reported include MRCR at the 16 K – 256 K range.

09 · Training

FP8 pretraining, incremental hybrid retrofit, fast-thinking post-train

The 2.6 family is not trained from scratch. Per the Ling-2.6-flash release post, the hybrid attention stack was reached by incremental training from the Ling-2.0 base — the GQA layers of Ling 2.0 were converted into the 1 : 7 MLA + Lightning Linear interleave through continued training. Post-training introduces a "Contextual Process Redundancy Suppression" reward to shorten chain-of-thought.

FP8 pretraining

Ling-1T base: 20 T+ high-quality, reasoning-dense tokens. End-to-end FP8 mixed precision; the predecessor release post reports ≤ 0.1 % loss deviation from BF16 across 1 T tokens of comparison, and a heterogeneous interleaved 1F1B pipeline that lifts utilization > 40 %.

Hybrid retrofit

Ling-2.6-flash release post: GQA attention layers of the Ling-2.0 base are converted into a 1 : 7 MLA + Lightning Linear hybrid through continued training, rather than a from-scratch redesign. The 2.6-1T model is the trillion-parameter sibling of this conversion.

Instruction / SFT

Exact SFT corpus composition is not enumerated in the Ling-2.6-1T card. The 2.6 family targets agent and code workflows — Claude Code, OpenClaw, OpenCode, and CodeBuddy compatibility is explicitly tested.

Fast-thinking RL

"Contextual Process Redundancy Suppression" reward during post-training — a reward shape that penalizes verbose chain-of-thought and rewards direct answers. The model card describes this as a "fast thinking" mechanism that compresses output tokens without losing task accuracy.

Underlying scaling law referenced by the Ling 2.0 release post: arXiv 2507.17702. Additional arXiv IDs linked from the Ling-1T repo: 2507.17634, 2510.22115.

12 · Unknowns & caveats

What is, and is not, in the published Ling-2.6-1T sources

The model card is short and prose-led. Many block-level dimensions that one would normally pull from config.json are not surfaced in the public README and have not been independently verified for this brief.

Source-inventory gaps

  • Layer count, hidden size, head count, head dim, intermediate sizes — not enumerated in the public Ling-2.6-1T card prose. The repo ships config.json and configuration_bailing_moe_v2_5.py; consult those directly before quoting numbers.
  • Routed-expert count, top-k, shared-expert count, routed scaling factor — not enumerated. The architecture tag bailing_hybrid and config class name bailing_moe_v2_5 place this in the same family as Ling-1T but the exact MoE topology is not re-stated.
  • Active parameter count (≈ 50 B) — taken from the Ling-1T predecessor release post which describes 1 T total / 50 B active / 1-in-32 sparsity. The Ling-2.6-1T card does not republish the active-parameter number; if the hybrid retrofit changed the per-layer FFN, the 50 B figure may be approximate.
  • Hybrid pattern position — the 1 : 7 MLA : Linear ratio is taken from the Ling-2.6-flash release post, which says the architecture was carried into the 2.6 family. The exact position of the MLA layer within each octet is not stated; this brief draws it as the last layer of each group of 8 by convention only.
  • Position encoding — RoPE base, YaRN factor, or NoPE behaviour are not described in the Ling-2.6-1T card. MRCR is evaluated up to 256 K but the mechanism for reaching that window is not enumerated.
  • Tokenizer / vocabulary — vocab size and tokenizer family are not stated in the prose. The tool-call parser at serve is qwen25 but that does not imply the underlying tokenizer.
  • Benchmark numbers — only SWE-bench Verified at 72.2 is published (community-uploaded HF leaderboard entry) and Artificial Analysis Intelligence Index of 34. Other benchmarks (AIME26, BFCL-V4, TAU2-Bench, Claw-Eval, PinchBench, MRCR, IFBench) are described qualitatively in the model card without numeric scores in the README itself.
  • Lineage assumptions — every figure inherited from Ling-1T or Ling-2.6-flash is labelled as such in the spec sheet; verify against the trillion-parameter card before quoting.
