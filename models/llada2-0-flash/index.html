LLaDA2.0-flash
a 100B-class MoE discrete diffusion language model

LLaDA2.0 scales discrete diffusion LLMs to frontier MoE sizes by converting a pretrained autoregressive checkpoint rather than training from scratch. Conversion is driven by a 3-phase block-level WSD (Warmup–Stable–Decay) schedule: progressively grow the block size, run full-sequence diffusion at scale, then anneal back to compact block diffusion. SFT and DPO follow. LLaDA2.0-flash is the 100B-parameter, ~6.1B-active instruction-tuned variant — built for parallel decoding at deployment.

§ 01

End-to-end inference pipeline

Prompt → masked sequence → iterative denoising through MoE transformer blocks → final tokens. Unlike an AR decoder, all positions are updated in parallel each step.
A · TOKENIZE
Prompt encoding
Chat-template the user turn and tokenize with the inherited tokenizer (vocab 157,184).
<|usr|>WhydoesCamus…<|gen|>
B · INIT
Masked sequence
Reserve gen_length positions and initialize the response as all-[MASK]. Optionally split into blocks of block_length.
prompt [M][M][M][M] [M][M][M][M]
C · DENOISE
Iterative denoising
T = steps rounds. Each round the MoE transformer predicts a clean token for every masked position in the current block in parallel.
prompt Sis-yph-[M]is [M]happy[M][M]
D · REMASK
Confidence remask
Low-confidence predictions are re-masked and resolved in the next round; high-confidence tokens are committed. Block advances when filled.
prompt Sis-yph-usis happybecause[M][M]
E · EMIT
Final sequence
Repeat across blocks until all positions are committed or eos_early_stop fires. Detokenize to the chat response.
prompt Sisyphusishappy becauseheownshisfate.
x0 ≔ [M,…,M] pθ(x0 | xt, c) block ∈ [1 … L] T = steps Parallel-decoded: every masked position in the active block is updated simultaneously each round.
§ 02

Block-level WSD training schedule

The diffusion block size is itself scheduled. Three phases — warm-up, stable, decay — match the abstract's "progressive block-size growth → full-sequence diffusion → revert to compact-size block diffusion".
Phase 1 · Warm-up
Progressive block growth
Start the AR-initialized network on small diffusion blocks, then progressively enlarge. The model "learns to diffuse" while preserving inherited autoregressive knowledge.
block: small → large inherits AR knowledge
Phase 2 · Stable
Full-sequence diffusion at scale
Operate at full-sequence block size for the large-scale training stretch. This is where the bulk of token budget is spent at the dLLM frontier.
block = full L ≈ 20T tokens (cumulative)
Phase 3 · Decay
Anneal to compact block
Decay back to a small, deployment-friendly block size. Compact blocks are what the released model uses at inference, and what the recommended block_length = 32 setting reflects.
block: large → small deploy-shape match
§ 03

MoE transformer internals

A sparse Mixture-of-Experts transformer block routes tokens to a small subset of experts. The activation ratio is the headline number — ~6% of parameters fire per token.
Single MoE transformer layer
Conceptual layout. Self-attention runs over every token; the FFN is replaced by a routed expert pool. Top-k, total experts, and shared-expert count are not specified in the listed sources.

§ 04

From AR checkpoint to instruction-tuned dLLM

Conversion — not training-from-scratch — is the paper's headline efficiency claim. A pretrained autoregressive MoE is reshaped into a diffusion LM, then aligned.
AR origin
Pretrained AR MoE
Inherit weights from a frontier autoregressive MoE in the Ling2.0 series. No costly from-scratch training.
~20T tokens (cumulative)
seed: AR causal LM
Phase 1 · Warm-up
Progressive block diffusion
Adapt AR weights to the diffusion objective starting with small blocks, then grow. Knowledge-inheritance principle: don't erase what the AR base already knows.

Full-sequence diffusion

Compact block diffusion

Diffusion decoding controls

Inference is exposed through a small set of knobs. Recommended defaults come from the model-card "Best Practices" block.
Generation parameters
Settings the call signature exposes in the model card's generate() example.
Stepssteps
Number of denoising rounds across the response. More steps → quality up, latency up.
32
Block lengthblock_length
Number of tokens denoised together per block. The recommended deployment shape and what Phase 3 anneals toward.
32
Generation lengthgen_length
Total response budget initialized as [MASK]. Sample call uses 512; model-card guidance: up to 32,768 for most queries.
512 — 32k
Temperaturetemperature
Sampling temperature over the diffusion head's distribution. Higher values can cause language mixing and modest quality loss.
0.0
EOS early stopeos_early_stop
Terminate once an EOS token is committed, even if gen_length isn't filled.
true
Remasking policyremask
How low-confidence tokens are demoted back to [MASK] between rounds. Described conceptually in the paper as confidence-based remasking; exact threshold rule is not exposed in the listed sources.
policy not specified in sources
Denoising rounds across one block
block_length = 32, steps = 32. Each row is one denoising round; columns are positions within the active block. Masked positions resolve, low-confidence positions get re-masked next round.
Masked Low-confidence (will remask) Committed Final block
Speed ↔ quality tradeoff. Both steps and block_length set at 32 mean roughly one denoising pass per output token on average, but every step still updates the whole block in parallel — that's where the latency advantage over AR decoding comes from. Lowering steps shortens latency at the cost of more residual [MASK] / lower-confidence tokens; raising it converges further.

§ 07

Caveats & unknowns

Facts the listed sources (arXiv 2512.15745 abstract + model card) do not state. Listed here verbatim instead of being guessed.
!

Not specified in the listed sources

Confirm against the v2 PDF or the released config.json before citing.
  • MoE topologyTotal expert count, top-k, shared-expert ratio, and load-balancing scheme.
  • Hidden dim / FFNPer-layer hidden size and per-expert FFN dimension are not stated on the model card.
  • Attention variantWhether attention is MHA, GQA, MLA, or something else — only head count (32) is given.
  • TokenizerTokenizer family (and BPE / SentencePiece details) inherited from the AR base is not named in these sources.
  • Remasking ruleThe exact confidence threshold / scoring function used to demote tokens back to [MASK] across denoising rounds.
  • Schedule valuesPhase boundaries, peak block size, learning-rate decay shape, and token budget per phase of the WSD curve.
  • AR base identityThe exact Ling2.0 checkpoint the flash variant is converted from is not pinned in the abstract or model card.
  • SFT & DPO dataDatasets, mix ratios, and preference-pair sources for the alignment stages.
  • DeploymentThroughput, KV / cache layout, recommended GPU configurations, and quantization recipes are out of scope for the listed sources.
  • AR-compatible modeWhether the released checkpoint can also be sampled left-to-right as an AR model is not asserted in the listed sources.
