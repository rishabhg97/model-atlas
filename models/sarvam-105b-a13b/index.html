02 · Decoder block

Inside one transformer layer

Sarvam-105B builds on the modern pre-norm decoder pattern. Two sub-modules per residual: an MLA attention block, then a sparse MoE feed-forward. RMSNorm gates each entry; the residual stream stays at width 4 096 from layer 1 to layer N.

Pre-norm block schematic

Sub-layer 1 — Attention residual
h = x + MLA( RMSNorm(x) )
Sub-layer 2 — Sparse-MoE residual
z = RMSNorm(h) g = softmax_top8( W_router · z ) // aux-loss-free balancing y_r = Σ_{i ∈ top-8} g_i · E_i(z) // routed experts (of 128) y_s = E_shared(z) // always-on shared expert x' = h + 2.5 · ( y_r + y_s ) // routed_scaling_factor
Each expert (SwiGLU MLP, dim 2 048)
E_i(z) = W_down_i · ( SiLU(W_gate_i · z) ⊙ (W_up_i · z) )
SwiGLU is the standard activation in this generation of Sarvam-style MoEs; the exact gate is not enumerated in the published model card, but the shape moe_intermediate_size = 2048 is.
03 · MLA attention

Multi-head Latent Attention with decoupled QK

Sarvam-105B's attention compresses the KV path into a single latent of width 576 per head, then re-projects on demand. Queries split into two independent channels — one carrying rotary position, one carrying no position at all. The split makes the rotary phase orthogonal to content, which is what makes YaRN-extrapolation to 128 K stable.

Per-head channel budget

Each head's 192-dim query is decoupled into a RoPE-rotated half and a no-position half; the value path is a separate 128-dim projection. The compressed KV latent (head_dim = 576) is what's actually cached.

04 · Positional encoding

RoPE and NoPE, sharing the same head

The 192-dim query head is concatenated from two independently-projected sub-vectors. Half is rotated by Rotary Position Embedding (and re-scaled by YaRN at long context); half is left untouched — "NoPE." Attention scores are the sum of two dot products, one positional and one content-only.

05 · Sparse Mixture-of-Experts

128 routed experts, top-8, plus one always-on shared expert

Every transformer-block FFN slot is replaced with a sparse MoE bank. The router picks the eight best experts for each token, multiplies their outputs by the softmax gate weights, sums in a shared expert that fires for every token, and rescales by 2.5. Auxiliary-loss-free balancing keeps expert utilization even without a separate load-balancing loss.

One routing event

Each square is one of 128 routed experts. For the token currently being processed, the router has picked 8 — they fire, weighted by their gate score. The shared expert (right) fires for every token.

06 · Long context

YaRN ×40 — how the model reaches 128 K

The native rotary period is extended by a factor of 40 using YaRN, putting the trained context at 128 K tokens. The model card recommends evaluating with a maximum context of 65 536 tokens. Inference engines ship MLA-specific kernels (flashmla in SGLang) to make the 576-dim KV latent affordable.

08 · Training pipeline

Pretraining → SFT → RL with a CISPO-inspired objective

The release blog discloses the high-level shape of the training pipeline. Pretraining used 12 T tokens across code, math, multilingual, and web data. The RL phase swaps standard clipped-surrogate methods for a group-relative objective inspired by CISPO and omits the reference-model KL anchor.

Pretraining

12 T tokens total. Mixture spans code, math, multilingual web data, and a heavy weighting on the 22 Indian languages and 12 scripts the tokenizer was built around. Training infrastructure was provided by Yotta on the IndiaAI Mission's government-backed GPU pool; NVIDIA contributed technical support.

SFT

Specific corpora, mixture weights, and SFT step counts are not reported in the model card or release blog. Chat template ships as a Jinja file in the repo.

RL alignment

Custom group-relative objective inspired by CISPO. KL-divergence against a reference model is omitted to avoid the optimization conflict with reward maximization. Trajectory staleness is bounded by capping age of sampled trajectories vs. policy updates. Reward shaping promotes structured reasoning, concise responses, and correct tool usage.

11 · Unknowns & caveats

Where the public sources stop

Facts we did not see in the model card, the release blog, or the public file listing. Nothing on this page is inferred from neighbouring model families; missing facts are left as such.

Not confirmed in source inventory

  • Exact layer count (num_hidden_layers). Public model card describes the block but does not enumerate depth. The configuration_sarvam_moe.py module exists in the HF repo but its contents were not parsed for this brief.
  • Number of attention heads (Q heads) and KV groups. MLA replaces the standard GQA grouping with a compressed latent; the head count is not published in the model card.
  • Vocabulary size. A custom tokenizer for 22 scheduled languages across 12 scripts is reported, but vocab size and BPE/Unigram details are not in the public model card.
  • Indic-language benchmark table. The release post claims a 90 % pairwise win rate on Indian-language benchmarks but does not publish the per-task comparison matrix in the public materials surveyed.
  • SFT and RL training corpora. RL objective (CISPO-inspired, no KL anchor) and reward shaping principles are reported. Specific datasets, mixture weights, and step counts are not reported.
  • Knowledge cutoff date. Third-party coverage reports ~June 2025; the model card itself does not list a cutoff.
  • Native (pre-YaRN) context window. Sarvam-30B reports a 32 K conversational window. For Sarvam-105B, only the extended 128 K (with YaRN ×40) is reported.
  • Activation function inside experts. SwiGLU is conventional for this generation but the public model card prose does not state it explicitly; configuration_sarvam_moe.py in the repo would be authoritative.
