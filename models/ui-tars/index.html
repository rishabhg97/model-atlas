UI-TARS-1.5-7B

An open-source multimodal agent that takes a screenshot and a natural-language instruction as input, reasons in a "Thought" trace, and emits structured GUI actions — clicks, drags, keystrokes, scrolls — with absolute pixel coordinates. Released by ByteDance Seed on 2025-04-16 as the 7B-parameter, generally available checkpoint of the larger UI-TARS-1.5 system.

02Inputs & preprocessing

UI-TARS-1.5-7B consumes one screenshot and one text instruction per agent step. Both go through Qwen2.5-VL's preprocessing pipeline; UI-TARS does not introduce new tokenizers or encoders here.

04Thought & action decoding

At inference, UI-TARS-1.5-7B autoregressively emits a "Thought" trace followed by exactly one Action call. The model card calls this "advanced reasoning enabled by reinforcement learning" and frames it as inference-time scaling — more reasoning tokens per step produces stronger decisions.

Default output format

Per step
# Raw assistant turn produced by the model
Thought: Click the button
Action: click(start_box='(100,200)')

Format and concrete example are taken verbatim from the official repo README's post-processing snippet. REPO

The parser maps the action string into a structured dict and then into executable pyautogui code for desktop environments. REPO

i
Thoughts vs. grounding. The GROUNDING prompt template suppresses the Thought trace and asks the model to emit only the Action line. Use it when you want to evaluate grounding capability in isolation or to minimise tokens per step. REPO

05Action vocabulary by platform

The repo ships three system-prompt templates in codes/ui_tars/prompt.py; each defines a different action vocabulary. The model is the same checkpoint — the prompt template constrains the legal verbs.

07Agent inference loop

UI-TARS-1.5-7B is wrapped in a perceive-think-act loop driven by an external runtime (the UI-TARS-desktop app for local desktops, Midscene for browsers, or any OSWorld-style runner).

08Pretraining & post-training

The HF model card frames UI-TARS-1.5 as "an open-source multimodal agent built upon a powerful vision-language model … integrating advanced reasoning enabled by reinforcement learning." Detail beyond that headline is mostly absent from the listed sources for the 7B checkpoint specifically.

Foundation

Stage 1

Pretrained vision-language model (Qwen2.5-VL family). UI-TARS-1.5-7B starts from this and does not retrain the VLM from scratch.

Agent SFT

Stage 2

The UI-TARS paper (arXiv 2501.12326) describes the foundational agent architecture and its native-agent training recipe. The 1.5 blog post identifies it as the foundation for 1.5.

not reported in the 1.5-7B card: exact GUI dataset scale, modality mix, screenshot domains, augmentations used for this checkpoint.

RL reasoning

Stage 3 · 1.5

The 1.5 release adds reinforcement learning that "allows the model to reason through its thoughts before taking action" and is what gives 1.5 its inference-time-scaling behaviour. HF

not reported RL algorithm (PPO / DPO / GRPO / other), reward shaping, environment scale, or whether the 7B checkpoint received the full RL pipeline of the larger 1.5.

!
Be careful with the 7B vs. 1.5 distinction. The README explicitly flags that "the released UI-TARS-1.5-7B focuses primarily on enhancing general computer use capabilities and is not specifically optimized for game-based scenarios, where the UI-TARS-1.5 still holds a significant advantage." Game benchmarks (Poki, Minecraft) are reported for UI-TARS-1.5, not 1.5-7B. REPO

What grounding actually measures

caveat

Grounding benchmarks isolate the screenshot-to-coordinate mapping. They do not measure planning, long-horizon execution, or environment dynamics.

A high ScreenSpotPro score is compatible with a much lower OSWorld score (UI-TARS-1.5-7B is 49.6 vs 27.5) because OSWorld requires 100 sequential decisions in a live environment, not one well-grounded click.

OSWorld step budget per README: 100 steps. ScreenSpotPro is a single-shot grounding metric.

12Limitations & unknowns

Two kinds of caveat: limitations the authors call out themselves, and facts the listed sources simply do not state.

Author-stated limitations

README
MisuseStrong GUI behaviour (incl. solving CAPTCHAs) could be repurposed for unauthorised automation; internal safety evals are described as "underway".
ComputeSubstantial resources needed for large tasks or extended gameplay.
HallucinationOccasional misidentification of UI elements or wrong inferences in ambiguous/unfamiliar environments.
Scope (7B)The 7B checkpoint targets general computer use; it is not specifically optimised for games — game scores in the README belong to the full UI-TARS-1.5 system.
